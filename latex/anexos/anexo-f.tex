Este anexo complementa la Tabla~1.4 (Cap.~1 §1.4) con un \textbf{ejemplo reducido} y un

\section{Ejemplo ampliado (fragmento de contrato)}

\subsection{Contrato OpenAPI (extracto ilustrativo)}

\begin{verbatim}
openapi: 3.0.3
info:
  title: Retail Payment Initiation API
  version: 1.0.0
paths:
  /payment-initiations:
    post:
      operationId: createPaymentInitiation
      summary: Initiate a payment order
      requestBody:
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/PaymentInitiation'
      responses:
        '201':
          description: Payment initiation created
  /payment-initiations/{paymentInitiationId}:
    get:
      operationId: getPaymentInitiation
      summary: Retrieve payment initiation status
      parameters:
        - name: paymentInitiationId
          in: path
          required: true
          schema:
            type: string
      responses:
        '200':
          description: Payment initiation details
components:
  schemas:
    PaymentInitiation:
      type: object
      required: [debtorAccount, instructedAmount, creditor]
      properties:
        debtorAccount:
          $ref: '#/components/schemas/AccountReference'
        instructedAmount:
          $ref: '#/components/schemas/Amount'
        creditor:
          $ref: '#/components/schemas/Party'
    Amount:
      type: object
      properties:
        value:
          type: number
        currency:
          type: string
          pattern: '^[A-Z]{3}$'
\end{verbatim}

\section{Nota metodológica}

El procedimiento completo de normalización, matching y cálculo de scores se documentará en el
Capítulo~4 (diseño metodológico). Este anexo satisface la solicitud docente de contar con un
ejemplo pequeño (Tabla~1.4) y uno ampliado (I5, PI-10) sin comprometer la decisión de equipo de
no fijar un piloto en la matriz de consistencia.
